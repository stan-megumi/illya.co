1.2 execution speed and memory usage

1.2.1 reasonable setup and usage of 64-bit unsigned integer

  • the following integer are all 64 bit unsigned integer, by loading npm install int64, and in fact, embedding the support for 64 bit unsigned integer is in the agenda of next version of javascript, Chromium version 57 and Mozilla already support it natively.
  • each battle might have n rounds, we focus on how to clear (清算) one round, then loop the process.
  • each round has TWO sides, each side has THREE on-screen places, and TWO bench places, total TEN places.
    • total of 10 characters data (noted as array) is loaded in one round randomly but then the position fixed in an array.
    • it suffices a 4-bit integer 0b1111 or 0xf (16 > 10+1 (nobody,death) ) to indicate character's array index.
    • it suffices a 64-bit integer (64 bit > 4bit * 10 places) to indicate the PlaceBar: who is standing on which place.
      0x87612345
      this is the current case: load right-side five characters from join to bench first, then load 3 enemies on the left side.
      0x12345
      this is the case where the left are all dead (leading zeros).
      0x34500
      this is the case the right-side won, but our first two characters dead.
      0xa987613245
      this is a 5v5 but the characters are loaded randomly.
      PlaceBar & 0xfffff00000
      2 cpu instructions to check if left side are all dead.
      PlaceBar & 0xfffff
      2 cpu instructions to check if right side are all dead.
    • if each player can charge only ONE Card skill, then there will be at most 10 cards
    • on OrderBar, there are at most (6+10)=16 (6 on-screen and 10 card) items
    • it suffices 9-bit integer 0o777 to indicate the order value as it is between 0 and 500.
    • on OrderValueBar, we need at most (6+10)*9=144 bits. it suffices THREE 64-bit integers to indicate the OrderValueBar, but you can add more as you want.
      0o0 0o0 0o 666 555 444 333 333 000
      (we insert the blank for better readability) this is the common case: 6 characters on-screen, with speed 438 (0o666), 365 (0o555), …, and 0 (0x00) the latest one.
      0o0 0o0 0o 666 555 444 333 333 444
      if the latest action cost 292 order value, use only one cpu instruction: OrderValueBar + 292.
      0o0 0o0 0o 666 555 444 444 333 333
      x= OrderValueBar & 0o777, y=(OrderValueBar & 0o777000)>>9 if x >= y is true then swap the two bits OrderValueBar & 0o777...7000 | (x<<9) | y, loop until false.
      0o0 0o0 0o 333 222 111 111 000 000
      OrderValueBar - 0oyyyyyy, ofc you generate 0oyyyyyy
    • in accordance of OrderValueBar, we create THREE 64-bit integers each indicating the array index of the OrderBar. as in the above example
      0o0 0o0 008 007 006 003 002 001
      notice bench is not on the OrderBar, now ready for character 1 in the array to move.
      0o0 0o0 008 007 006 003 002 001
      character 1 acts and action cost 292 order value.
      0o0 0o0 008 007 001 006 003 002
      now it is 0o02, the second in the array index's turn to act. each time the abve x >= y is true, 01 left shift (in human sense not <<).
    • summary
      • ONE 64-bit integer to represent the PlaceBar
      • THREE 64-bit integers to represent the OrderBar
      • THREE 64-bit integers to represent the OrderValueBar
      • usually we only need only one integer of OrderBar and OrderValueBar
      • with minimum cpu instructions (to be optimized) to achieve the order clear system.
      • but as you can see, 64-bit integer can hold 16 characters in 16 places in PlaceBar
      • more getter functionalities needed, but all can be implemented within several cpu instructions.

1.2.2 usage of mask for toggling and setting

suppose we have 

public enum eStateAbnormal
{
        Confusion,
        Paralysis,
        Poison,
        Bearish,
        Sleep,
        Unhappy,
        Silence,
        Isolation,
        Num
}
  • we suffice only one integer s to indicate eStateAbnormal rather than a vector
    s|=(1<<eStateAbnormal.Confusion)
    to toggle on Confusion
    s|=Confusion
    more if we write var Confusion = (1<<eStateAbnormal.Confusion)
    s&=Confusion
    no more Confusion
    s|=Confusion|=Poison
    toggle on Confusion and Poison
    s & Confusion
    if s suffer from Confusion
  • effectively storing buff ,status and their parameters
    • now suppose we have less than 64 buffs (status), and with parameters with it.
    • as it is a sparse vector, storing as a full size-64 vector is time and memory consuming
    • we can append effective parameter to an empty vector sv.
    • suppose we have Confusion and Poison, we only have an sv of two elements.
    • the parameter index for Confusion (suppose there is one): sv[counting 1 to the right of Confusion bit.]
  • At Last
    • we need ONE integer of buff, and a vector of suffered buff parameters to store buff effects
    • we need ONE integer of status, and a vector of necessary status parameters to store status effects
    • we need ONE integer of elementType, but we may have MULTIPLE Fire and Water element on a single character
    • we need ONE integer of occupation, but we may have MULTIPLE Knight and Magician occupations on a single character
    • the methods for checking, toggling attributes and fetching parameters should be within several cpu instructions.

1.2.3 splitting up skills, characters and players

  • suppose we have 8 bit 0xff to indicate the index from 255 total skills, loaded once, then never copied to anywhere or change the sequence
  • for each character we can have one 64 bit 8 skill * 8 bit=64bits integer to show 8 skills he/she got from the skills.
  • for each character we can have one 64 bit 8 skill * 8 bit=64bits integer to show 8 skills's usage.
  • for each player we can have one 64 bit integer to show his/her information set. e.g. 
    enum InfoSetType {
   NONE=0;
   AVAILABLE_CHARACTER=1<<1; (achara)
   AVAILABLE_SKILLS=1<<2; (askill)
   ENEMY_HP=1<<3; (ehp)
   FRIEND_HP=1<<4; (fhp)
  }
  • everytime the player need to quest the world what he knows, including available actions, then pick one move.

1.3 functionality

1.3.1 information mask

Player in the following functionalities is in the name of a human-being player. imagine he/she suffered a buff that:

  • the player may not know his/her teammates information.
  • the player may not know opponent's attributes.
  • the player may not know which character he/she is controlling.
  • the player may not know the the number of opponents, they are dead or not.
  • upon damage, the player may not know the opponent's action.
  • upon hiding and shuffling the available skill cards, the player may not know which skill he/she is picking, . (this is different from suffering chaos as the case he/she was passively given a random skill action by the world, rather than actively picking an unknown one)
  • invisiable player
  • any data or information in this game may/may not be reached by any player based on the player's information mask.
  • use your imagination to create interesting skill effects

1.3.2 multi-player

  • player on both sides
  • multi-player on the same side
  • each player's gui is different based on its information set.

  • this function return the *World.Player[World.Environment.CurrentPlayer].InfoSet*
  • this function illustrate a getter method for getting a data without checking if this data is available to current player.

  • this function can be called to get the index of the least hp character of our side or opponent side.
